# ADR-0004: Use AWS EC2 for Cloud Hosting

**Status:** Accepted

**Date:** 2026-06-29

# Context

[ADR-0003](0003-use-terraform.md) established that cloud infrastructure for staging and production will be managed with Terraform on AWS.

The prototype needs a compute platform to run WordPress, CiviCRM, Nginx, and MariaDB. Several hosting options are viable for a small county party site:

* **AWS Lightsail** — simplified AWS VPS with predictable pricing
* **AWS EC2** — full-featured virtual servers within a customizable VPC
* **DigitalOcean** or **Linode** — independent cloud providers with straightforward VPS offerings
* **Managed WordPress hosting** — platform-as-a-service with limited CiviCRM control

The long-term vision is a reusable platform that other Wisconsin county Democratic parties could adopt. See [architecture.md](../architecture.md). If successful, CoCoDems may grow from a single-county prototype into a statewide deployment model with dozens of independent county instances.

Initial traffic and resource requirements are modest. The hosting decision must nevertheless account for growth, integration needs, and who will operate the infrastructure.

# Decision

Use **AWS EC2** as the primary compute platform for staging and production.

The initial deployment may be a single EC2 instance (or a small fixed set of instances) running Docker or native LAMP/LEMP services. Architecture will be designed so additional AWS services can be adopted incrementally without changing cloud providers.

# Rationale

## Advantages

* **Control.** EC2 offers full control over instance type, storage, networking, security groups, and operating system configuration. This suits a WordPress + CiviCRM stack that requires custom PHP extensions, cron jobs, and backup workflows.
* **AWS ecosystem.** EC2 integrates natively with services that hosting vendors do not provide as a cohesive platform:
  * **S3** — database and file backups, static asset storage
  * **RDS** — managed MariaDB with automated backups and Multi-AZ failover
  * **Lambda** — event-driven tasks (webhooks, scheduled sync jobs, image processing)
  * **CloudFront** — CDN for static assets and caching
  * **WAF** — web application firewall for public-facing endpoints
  * **IAM roles** — least-privilege access without long-lived credentials on instances
  * **Route 53** — DNS management
  * **CloudWatch** — monitoring, logging, and alerts
* **Growth path.** If the CoCoDems CRM develops into a statewide platform with dozens of county parties, EC2-based architecture can scale without a provider migration:
  * Auto Scaling Groups
  * Application Load Balancers
  * RDS Multi-AZ
  * ElastiCache
  * EFS
  * S3
  * CloudFront
  * WAF
  * Lambda
  * IAM roles
  * VPC peering
  * Transit Gateway
* **Terraform alignment.** EC2, VPC, RDS, S3, and IAM are first-class resources in the AWS Terraform provider, supporting the reproducible infrastructure goals in [ADR-0003](0003-use-terraform.md).
* **Maintainer expertise.** The project's primary infrastructure maintainer (Sheldon Rampton) already has professional experience with EC2, IAM, Route 53, CloudWatch, Linux administration, security groups, RDS, and Terraform. Choosing EC2 reduces ramp-up time and operational risk during the prototype phase.

## Alternatives considered

| Alternative | Why not chosen |
|-------------|----------------|
| **AWS Lightsail** | Simpler and cheaper for a single small site, but offers less networking flexibility and a narrower growth path into the broader AWS services listed above. A reasonable choice for teams without existing AWS expertise. |
| **DigitalOcean** | Straightforward VPS hosting and good documentation, but would require adopting a separate provider ecosystem (Spaces, managed DB, etc.) and rewriting Terraform modules against a different provider. |
| **Linode (Akamai)** | Similar trade-offs to DigitalOcean — viable for a single site, but weaker alignment with the AWS + Terraform stack already chosen. |
| **Managed WordPress (WP Engine, etc.)** | Reduces administration burden but limits CiviCRM installation, custom plugin deployment, and infrastructure-as-code reproducibility. |

## Disadvantages

* **Operational complexity.** EC2 requires more setup than Lightsail or a simple VPS — VPC design, security groups, key management, patching, and monitoring are the operator's responsibility.
* **Cost unpredictability.** EC2 pricing depends on instance type, storage, data transfer, and attached services. Costs must be monitored; a misconfigured environment can exceed Lightsail's fixed monthly price.
* **AWS coupling.** Deep investment in AWS-specific services (RDS, Lambda, CloudFront) increases the cost of migrating to another cloud vendor later.
* **Volunteer handoff.** Future maintainers need Linux and AWS skills, or must rely on AI-assisted operations and thorough documentation. Lightsail or managed hosting would lower the skill floor.

## Note on expertise

If the project did not have existing AWS and Terraform expertise to draw upon, **AWS Lightsail** would be a strong alternative for the prototype: predictable pricing, simplified networking, and a gentler learning curve for volunteer operators.

The choice of EC2 is partly a pragmatic match between platform ambition and the skills available to the project today.

# Consequences

* Terraform modules in `infra/terraform/` will target EC2 (and associated VPC, security group, and IAM resources) rather than Lightsail instances.
* The prototype may begin with a **minimal** topology — e.g., one EC2 instance with local or RDS-hosted MariaDB — rather than deploying the full scaling stack on day one.
* Backup strategy should use **S3** for off-site database and file storage, as described in [architecture.md](../architecture.md).
* DNS will be managed in **Route 53** (or documented if an external registrar remains authoritative).
* Monitoring and alerting will use **CloudWatch** at minimum; alarms for disk, CPU, and backup failures should be defined in Terraform.
* Cost budgets and monthly review should be established before production launch.
* Documentation and runbooks must explain EC2 operations clearly enough that a volunteer without AWS experience can perform routine tasks with AI assistance.
* Each county deployment remains independent (separate database, website, administrators, and backups) per [architecture.md](../architecture.md); EC2 enables this without mandating a shared multi-tenant architecture.
